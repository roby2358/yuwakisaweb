# Agent Instructions: Port SimHospital Simulation Core to Java

## Goal
Port the simulation engine from the Go-based SimHospital project (in the `simhosp/`
directory) to a self-contained Java library. The deliverable is a Maven project
containing a `HospitalSimulator` class that exposes a `nextMessage()` method
returning well-formed HAPI HL7v2 `Message` objects. This library will be embedded
in a performance testing framework — it has no server runtime, no HTTP layer, and
no transport concerns.

---

## Step 1: Read and Understand the Source

Before writing any code, read the following from `simhosp/`:

- `README.md` — overall architecture
- `config/pathways/` — all pathway YAML/JSON files; understand the event model
  (admission, discharge, order, result, transfer, etc.)
- `pkg/pathway/` — how pathways are parsed and executed
- `pkg/message/` — how HL7v2 messages are constructed from events
- `pkg/patient/` — how patient state is modelled and evolves through a pathway
- `pkg/clock/` — the simulation clock abstraction
- `pkg/simulator/` — the top-level simulation loop and event queue

Do not begin writing code until you have read all of the above.

## Step 4: Core Design

Implement the following classes and interfaces exactly as specified. Where Java 21
features (records, sealed interfaces, virtual threads) are a natural fit, use them.

---

### `SimulationClock` (interface)

```java
public interface SimulationClock {
    /** Returns the current simulation time. */
    Instant now();

    /**
     * Advances the clock to the given target time.
     *
     * WallClock: blocks the calling thread until wall time reaches or exceeds
     * target. This enforces real-time pacing.
     *
     * SyntheticClock: sets internal time to target and returns immediately.
     * This allows faster-than-realtime simulation.
     *
     * Calls with a target earlier than now() must be treated as a no-op —
     * the clock must never go backwards.
     *
     * HospitalSimulator is single-threaded; implementations do not need to
     * be thread-safe. One SimulationClock instance per HospitalSimulator.
     */
    void advanceTo(Instant target) throws InterruptedException;
}
```

Provide two implementations:

**`WallClock`** — `now()` delegates to `Instant.now()`. `advanceTo(target)` calls
`Thread.sleep()` in a loop until `Instant.now().compareTo(target) >= 0`.

**`SyntheticClock`** — holds a plain `Instant currentTime` field. `now()` returns
it. `advanceTo(target)` sets `currentTime = target` if `target` is after the
current value, otherwise does nothing. Constructed with a start epoch:
```java
new SyntheticClock(Instant.parse("2024-01-01T00:00:00Z"))
```

---

### `PatientState` (mutable class — not a record)

Holds the full mutable state of one simulated patient. `ScheduledEvent` holds a
live reference to this object, not a snapshot — mutations to `PatientState` after
scheduling (e.g. advancing `nextEventIndex`) are visible in the queue.

```java
public class PatientState {
    String mrn;                           // generated unique ID
    String familyName;
    String givenName;
    LocalDate dateOfBirth;
    String gender;                        // "M", "F", or "U"
    String pathwayName;
    int nextEventIndex;                   // index of next event in Pathway.events()
    Instant admissionTime;
    String ward;
    String bed;
    Map<String, OrderDetails> openOrders; // orderId → OrderDetails
}
```

`OrderDetails` is an immutable record:
```java
public record OrderDetails(String orderCode, String description) {}
```

---

### `PatientGenerator`

Generates new `PatientState` instances with randomised demographics. Accepts a
`Random` instance (seeded from `SimulatorConfig`) so output is reproducible.
Generates synthetic names, DOBs, and gender distributions — do not use real
patient data.

### `ScheduledEvent` (record)

Used internally by `HospitalSimulator`'s `PriorityQueue`. Place in the `simulator`
package. `patient` is a reference to the live, mutable `PatientState` — not a
copy. Mutations to `PatientState` after scheduling are reflected immediately.

```java
public record ScheduledEvent(
    Instant scheduledTime,
    PatientState patient,
    PathwayEvent event
) implements Comparable<ScheduledEvent> {

    @Override
    public int compareTo(ScheduledEvent other) {
        return this.scheduledTime.compareTo(other.scheduledTime);
    }
}
```

### `MessageFactory`

```java
public class MessageFactory {
    /**
     * Constructs a HAPI Message from the given event and patient state.
     * Returns Optional.empty() for DELAY events (no message is produced).
     *
     * Use HAPI's typed message classes — do not build HL7v2 strings manually.
     *
     * To encode a Message to a String (e.g. in tests or the smoke test), use:
     *   new PipeParser().encode(message)
     * Do NOT call encode() inside this class.
     */
    public Optional<Message> build(PathwayEvent event, PatientState patient,
                                   Instant eventTime) { ... }
}
```

Event-to-message-type mappings:

| PathwayEventType | HAPI Message Class | Notes                          |
|------------------|--------------------|--------------------------------|
| `ADMISSION`      | `ADT_A01`          |                                |
| `DISCHARGE`      | `ADT_A03`          |                                |
| `TRANSFER`       | `ADT_A02`          |                                |
| `ORDER`          | `ORM_O01`          | ORC-1 = "NW" (new order)       |
| `CANCEL_ORDER`   | `ORM_O01`          | ORC-1 = "CA" (cancel)          |
| `RESULT`         | `ORU_R01`          |                                |
| `OBSERVATION`    | `ORU_R01`          | Typically vital signs          |
| `DELAY`          | *(none)*           | Return `Optional.empty()`      |

Populate the following segments from `PatientState` on every message:
- **MSH**: sending app, sending facility, datetime from `eventTime` formatted as
  `yyyyMMddHHmmss`, message control ID (UUID), processing ID "P", version "2.5"
- **PID**: MRN in PID-3, patient name in PID-5, DOB in PID-7, gender in PID-8
- **PV1** (where applicable): patient class, assigned location (ward/bed),
  admission datetime

For fields not tracked in `PatientState` (e.g. attending physician name, lab
result values), generate plausible synthetic values using the seeded `Random`
from `SimulatorConfig`. Lab values should fall within clinically normal ranges.

### `HospitalSimulator` (main class)

```java
public class HospitalSimulator {

    public HospitalSimulator(SimulatorConfig config) { ... }

    /**
     * Returns the next HL7v2 message in simulation-time order.
     *
     * Runs entirely on the calling thread — there is no background thread.
     * To run multiple parallel streams, create multiple HospitalSimulator
     * instances (one per thread) with the same config.
     *
     * With WallClock: blocks until the event's scheduled time is reached via
     * clock.advanceTo(), then returns the message. If messageRatePerSecond is
     * set, additionally sleeps to enforce the configured rate.
     *
     * With SyntheticClock: advances the clock to the event's scheduled time
     * and returns immediately — no blocking occurs. messageRatePerSecond is
     * ignored.
     */
    public Message nextMessage() throws InterruptedException { ... }
}
```

**Internal execution model:**

`nextMessage()` runs entirely on the calling thread:

1. Peek at the head of the `PriorityQueue<ScheduledEvent>` (earliest event)
2. Call `clock.advanceTo(event.scheduledTime())` to pace delivery
3. Pop the event, invoke `MessageFactory.build()`
4. If the event produces a message, advance the patient's `nextEventIndex`.
   If the patient's pathway is now complete, replace them with a new patient
   to keep the pool at its configured size. Re-enqueue the next event for this
   patient (or the replacement), then return the message.
5. If the event is a `DELAY` (no message), re-enqueue the patient's next event
   and loop back to step 1.
6. If `messageRatePerSecond` is set and `WallClock` is in use, apply a
   `Thread.sleep` throttle after step 2 based on elapsed wall time.

## Step 6: Tests

Write JUnit 5 unit tests covering the following:

**`PathwayLoaderTest`**
- Successfully parses all YAML files in `simhosp/config/pathways/`
- Each loaded pathway has at least one event
- All event types map to recognised `PathwayEventType` values

**`MessageFactoryTest`**
- For each non-DELAY `PathwayEventType`, `build()` returns a non-empty `Optional`
- The returned `Message` round-trips through `new PipeParser()` without throwing
- MSH-7, PID-3, PID-5 are non-empty on every message

**`SyntheticClockTest`**
- `advanceTo(t)` results in `now()` returning `t`
- Calling `advanceTo()` with a time earlier than `now()` is a no-op (clock does
  not go backwards)

**`HospitalSimulatorTest`**
- With `SyntheticClock`, 1000 consecutive `nextMessage()` calls return messages
  whose MSH-7 timestamps (parsed as `yyyyMMddHHmmss`) are in non-decreasing order
- A pathway containing only `ADMISSION` then `DISCHARGE` produces an `ADT_A01`
  before an `ADT_A03` for the same MRN (verify via PID-3)
- With `seed` set, two separate `HospitalSimulator` instances with identical
  config produce the same sequence of MSH-7 / MSH-9 values for the first 100
  messages

## Step 7: Smoke Test

After all tests pass, verify end-to-end with the following main method:

```java
var config = SimulatorConfig.builder()
    .pathwayDirectory(Path.of("simhosp/config/pathways"))
    .concurrentPatients(10)
    .clock(new SyntheticClock(Instant.parse("2024-01-01T00:00:00Z")))
    .seed(42L)
    .build();

var sim = new HospitalSimulator(config);
var parser = new PipeParser();

for (int i = 0; i < 100; i++) {
    try {
        Message msg = sim.nextMessage();
        String encoded = parser.encode(msg);
        System.out.println(msg.getName() + " | " + encoded.substring(0, 60));
    } catch (HL7Exception e) {
        throw new RuntimeException("Failed to encode message at index " + i, e);
    }
}
```

Expected output: 100 lines of mixed `ADT_A01`, `ADT_A02`, `ADT_A03`, `ORM_O01`,
and `ORU_R01` message names with no exceptions thrown.

## Constraints

- **Java 21+**: use records for immutable data classes
- **No frameworks**: plain library JAR only — no Spring, Quarkus, or any
  application container
- **No transport**: no MLLP, no sockets, no HTTP — the caller handles all of that
- **Single-threaded per instance**: `HospitalSimulator` is not thread-safe by
  design. To parallelise, the caller creates one instance per thread. Do not add
  synchronisation inside the simulator
- **Reproducibility**: when `seed` is set in `SimulatorConfig`, identical configs
  must produce identical message sequences across JVM runs
- **No background threads**: `HospitalSimulator` must not start any threads
  internally. All work happens on the calling thread inside `nextMessage()`
